# Висновок за модулем: Time & Space Complexity

У цьому блоці я поглибив свої знання щодо аналізу ефективності алгоритмів, що є критично важливим для написання оптимізованого коду.

## 1. Часова складність (Time Complexity)
Я пригадав, як оцінювати час виконання алгоритму залежно від обсягу вхідних даних ($n$). Основний акцент було зроблено на **Big O notation** (О-нотація), яка описує найгірший сценарій виконання.

* **Ітеративні підходи:** Аналіз циклів (`for`, `while`). Якщо цикл проходить по масиву один раз — це $O(n)$, якщо маємо вкладені цикли — $O(n^2)$.
* **Рекурсивні підходи:** Розрахунок складності через дерево рекурсії. Наприклад, для двійкового пошуку складність становить $O(\log n)$, а для рекурсивного обчислення чисел Фібоначчі без оптимізації — $O(2^n)$.

## 2. Просторова складність (Space Complexity)
Окрім часу, я навчився оцінювати витрату оперативної пам'яті:
* **Auxiliary Space:** додаткова пам'ять, яку алгоритм використовує (не враховуючи вхідні дані).
* **Стек рекурсії:** Важливо пам'ятати, що кожен рекурсивний виклик додає фрейм у стек, що може призвести до $O(n)$ просторової складності навіть без створення нових масивів.



## 3. Порівняльна таблиця складностей
Для швидкого орієнтування я закріпив розуміння основних класів складності:

| Нотація | Назва | Приклад |
| :--- | :--- | :--- |
| $O(1)$ | Constant | Звернення до елемента масиву за індексом |
| $O(\log n)$ | Logarithmic | Двійковий пошук |
| $O(n)$ | Linear | Перебір елементів списку |
| $O(n \log n)$ | Linearithmic | Ефективні алгоритми сортування (Merge Sort) |
| $O(n^2)$ | Quadratic | Вкладені цикли (Bubble Sort) |

**Результат:** Тепер я можу не просто писати робочий код, а й обґрунтовано обирати найбільш ефективний підхід, виходячи з обмежень по пам'яті та часу.
